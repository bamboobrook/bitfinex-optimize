# Bitfinex Lending Path Ranking Design

## 背景

当前程序的目标，是为 Bitfinex 自动放贷机器人提供 `result`，让外部机器人按 `rank1-6` 决定挂单策略。现有系统已经具备：

- 历史 funding 数据下载
- 多模型利率预测
- 虚拟订单验证
- 执行率闭环回灌
- 市场流动性打分

但最近线上表现已经暴露出明显偏差：

- `rank1` 会出现高利率但远离当前市场的组合
- 最近 12 小时内经常无法成交
- 中长周期高溢价组合在排序中过于靠前
- 内部闭环验证逻辑与外部机器人真实挂单路径不一致
- 流动性信号存在滞后，尤其在 `fUST` 上容易高估短时机会

本次设计的目标，不是简单提高“即时成交率”，而是把内部评分目标对齐到外部机器人的真实执行路径，提升 `rank1` 的最终价值判断质量。

## 现状问题

### 1. 排序目标与真实执行链路错位

当前推荐主要围绕“单个静态报价值不值得挂”展开，但外部机器人真实行为是：

1. 按 `rank1-5` 初始固定利率挂单
2. 每小时下降挂单利率 `1%`
3. 持续 `6` 小时
4. 若未成交，转为 `FRR-120d` 再挂 `6` 小时
5. 若仍未成交，转为 `rank6`

因此，当前系统如果只优化“这一口固定报价的执行概率”，会高估那些起始利率很高、但最终大概率会掉进 `FRR` 或 `rank6` 兜底的组合。

### 2. `rank1` 对高利率假机会识别不足

线上最近结果已经出现：

- `fUSD-20d` 当前市场约 `5.5%`，预测却报到 `14.88%`
- 这类组合仍能获得较高排序位置

这说明当前的：

- `weighted_score`
- `calibrated_execution_probability`
- 风险降权

还不足以压制“高溢价但低真实路径价值”的候选。

### 3. 闭环标签没有贴合真实业务

当前虚拟单验证更接近“固定利率在 24/48/72h 静态窗口里是否成交”，但真实业务更关心：

- 前 `6h` 固定利率阶段值不值得先试
- `6-12h` 回退 `FRR-120d` 后的最终收益是否还能接受
- 最终是否只能依赖 `rank6`

如果标签定义不对，后续执行率回灌再多，也只是在优化错误目标。

### 4. 流动性信号不够快

用户要求流动性必须能反映最近市场变化，不能明显滞后。当前系统虽然已经有：

- 历史执行率
- funding book 信号
- volume ratio

但仍偏慢，尤其对 `fUST` 的短时爆发/退潮识别不够敏捷。

### 5. 币种优先级与业务偏好需要明确编码

用户已明确：

- 默认 `fUSD` 优先
- `fUST` 只有在收益和流动性都明显占优时，才允许抢到更高推荐位
- `fUST` 的中长周期默认不应轻易上位

这部分目前虽已有权重倾斜，但仍不够业务化。

## 用户确认后的业务原则

本轮设计以以下明确规则为准：

1. `result` 输出格式不能改
2. `result` 中 `rank6` 固定为 `fUSD-2d`
3. 外部机器人会随时请求策略，但内部 API 目前每 `2` 小时跑一轮
4. 外部机器人真实挂单链路：
   - 优先使用 `rank1-5`
   - `rank1` 资金 `60%`
   - `rank2-5` 各 `10%`
   - 每小时降利率 `1%`
   - 固定利率阶段持续 `6h`
   - 若未成交，转 `FRR-120d` 再挂 `6h`
   - 若仍未成交，转 `rank6`
5. 若总资金低于 `1500`，则全部按 `rank1` 挂单
6. 若 `120d` 预测年利率低于 `12%`，外部机器人会自动回退到 `FRR-120d`
7. 用户的主目标是：
   - 不是尽快成交
   - 而是避免在低利率时把资金锁入低收益长单
   - 优先争取更高的最终路径收益
8. `fUSD` 默认优先
9. `120d >= 12%` 时，长周期应被强优先

## 目标

### 主目标

把推荐目标从“静态报价即时可成交性”改成：

`真实执行路径的预期最终价值最大化`

即，系统要优先挑出“作为起始策略最值得先挂”的组合，而不是“眼下最容易立刻成交”的组合。

### 次目标

- `rank1` 不再频繁给出远离市场的假高利率
- 流动性信号更快，能反映最近 `2h/6h/12h` 的变化
- `fUSD` 在默认场景中更稳定地主导高位推荐
- `fUST` 只在短时爆发且流动性确认时上位
- `120d >= 12%` 时显著优先
- 内部验证逻辑与外部真实挂单行为对齐

## 非目标

本次不做以下变更：

- 不改变 `/result` 的 JSON 结构
- 不改变 `rank6 = fUSD-2d`
- 不重写外部机器人
- 不改外部兑换规则
- 不拆出新的独立服务
- 不把整个系统重构成多服务架构

## 方案选择

比较过三条路径后，本次采用：

`路径仿真排序` 为主，叠加低风险评分保护

不采用“纯调权重”作为主方案，因为它治标不治本；也不采用“双层新决策服务”，因为会增加不必要实体和维护成本。

## 总体设计

### 设计核心

每个 `(currency, period)` 候选，不再只产出一个静态推荐分，而是先生成基础预测，再做一层“真实执行路径仿真”，得到最终排序所需内部评分。

外部看到的结果仍然是现有格式；变化只发生在内部 `rank1-5` 的选择逻辑。

### 统一后的内部目标

系统内部所有关键环节都围绕同一目标展开：

- 预测阶段：预测这条起始路径值不值得先挂
- 验证阶段：验证这条路径最终是否真的值
- 重训阶段：学习什么样的起始路径最终更优

这会替代当前“预测看静态报价，验证看长窗口成交，排序再做一层经验调权”的松耦合方式。

## 预测阶段设计

### 基础预测层

保持现有能力：

- 预测利率
- 预测成交概率
- 历史执行率校准
- 市场锚定
- 流动性估计

这部分依然由现有模型和已有逻辑提供原始输入，不推翻模型基础设施。

### 路径仿真层

在基础预测之后，增加路径级内部指标：

- `expected_path_value`
- `expected_fill_0_6h`
- `expected_frr_fallback_value`
- `expected_rank6_fallback_cost`
- `expected_terminal_mode`
- `final_rank_score`

这些字段用于内部排序与落库诊断，不进入外部 `result` 结构。

### 路径解释

每个候选的内部路径语义为：

1. 以当前推荐利率作为起始固定利率
2. 模拟前 `6h` 每小时降 `1%` 的收益与成交机会变化
3. 若固定阶段失败，模拟回退到 `FRR-120d` 的收益价值
4. 若仍失败，计入最终回退 `rank6` 的机会成本

这样 `rank1` 排序将真正考虑：

- 这笔资金先试该组合是否划算
- 若失败，后续回退会损失多少
- 是否容易把资金浪费在本就不值得尝试的起始路径上

## 新的排序评分结构

最终推荐分定义为：

`final_rank_score = path_value_score × liquidity_multiplier × currency_regime_multiplier × safety_multiplier`

### 1. `path_value_score`

这是主分，表示真实执行路径下的预期最终价值。

重点体现：

- 起始固定利率越高越好
- 周期越长越好
- 若 `120d >= 12%`，强优先 `120d`
- 如果路径大概率会掉到 `FRR` 或 `rank6`，则主分要显著回落

### 2. `liquidity_multiplier`

用于表达当前市场是否真的支持这条路径。

设计原则：

- 快信号主导
- 慢信号只做校正

主要参考：

- 最近 `2h / 6h / 12h` 的市场活跃变化
- 最近几轮 funding book 的非 `2d` 深度
- 近短时历史成交跟随能力
- `7d/14d` 历史执行率
- 成交量收缩或爆发

### 3. `currency_regime_multiplier`

体现币种优先级和业务偏好。

规则：

- `fUSD` 默认优势
- `fUST` 只有在收益和流动性都明显占优时，才允许抢更高位
- `fUST` 的中长周期默认更保守

### 4. `safety_multiplier`

作为最后保险层，压制高溢价假机会。

重点惩罚：

- 预测利率相对当前市场溢价过高
- 同类组合最近连续失败
- 该组合长期积压 `PENDING`
- 大概率只是“先挂高价，最终还是回退”

## 流动性设计

### 设计目标

流动性必须能更快反映真实市场，而不是只依赖滞后的历史执行率。

### 快慢结合策略

#### 快信号

- 最新一轮及最近几轮实时 funding book 深度
- 最近 `2h/6h/12h` 成交量变化
- 短时间内市场利率位置和成交可跟随性

#### 慢信号

- `7d/14d/30d` 历史执行率
- 历史失败价差
- 长期样本稳定性

#### 聚合原则

- 快信号用于决定是否允许候选上高位
- 慢信号用于限制过度追逐短时假爆发

### 对 `fUST` 的特殊策略

- 默认保守
- 只有快信号和收益同时明显变强，才提升排序权重
- 不允许仅因旧的高执行率残留而抢占高位

## 验证阶段设计

### 问题

当前验证主要判断单个固定利率在 `24/48/72h` 窗口里的静态成交结果，这与真实机器人行为不一致。

### 新的验证语义

验证应尽量贴近真实路径，按阶段理解订单价值：

- `stage1_fixed_0_6h`
- `stage2_frr_6_12h`
- `stage3_rank6_after_12h`

### 设计原则

不新增服务，不改变对外接口，只增强内部验证标签与诊断记录。

后续训练时，模型要学的是：

`这个推荐作为起手策略，整条路径最终值不值`

而不是：

`这口固定利率在长窗口内最后是否成交`

## 落库设计

### `prediction_history`

新增内部诊断字段，用于记录本轮预测的路径级判断：

- `path_value_score`
- `stage1_fill_score`
- `stage2_frr_value`
- `terminal_fallback_penalty`
- `fast_liquidity_score`
- `currency_regime_state`
- `final_rank_score`

### `virtual_orders`

新增或补齐与真实路径有关的验证信息：

- 早期固定阶段是否合理
- 回退阶段价值
- 最终回退模式
- 路径级标签

### 原则

- 仅扩展内部表结构
- 不影响外部 `result`
- 为后续回测和重训提供可追溯证据

## 训练数据与重训设计

### 训练目标修正

重训不应继续只盯着“静态报价的最终执行率”，而要更多吸收路径级反馈。

训练样本要能反映：

- 起手固定高利率是否真的值得先挂
- 前 `6h` 固定阶段能否保住价值
- 回退到 `FRR` 后是否仍具备吸引力
- 最终落入 `rank6` 是否意味着起手策略本来就不理想

### 重训策略

保留现有重训调度器框架，仅修改其所依赖的反馈信号和样本字段，不重写整个训练系统。

## `120d >= 12%` 规则

该规则必须被内部显式编码，而不是只依赖外部机器人兜底。

设计要求：

- 当 `120d < 12%` 时，内部排序应自动降低其优先级，避免推荐与外部回退逻辑发生错位
- 当 `120d >= 12%` 时，长周期高收益应被强优先
- 该逻辑要同时影响：
  - 主路径价值分
  - 推荐排序
  - 后续回测统计

## 对外输出约束

以下内容必须严格保持：

1. `/result` 结构不变
2. `rank6` 固定为 `fUSD-2d`
3. 现有外部机器人读取字段方式不变
4. API 接口语义不变

内部可以新增更多诊断字段到数据库，但不把这些内部变化泄漏到对外接口。

## 服务运行方式设计

### 目标

把当前临时式运行方式切换为稳定的后台守护方式。

### 方案

采用 `systemd` 用户服务管理 API：

- 开机自启动
- 异常自动重启
- 后台运行
- 可查看运行状态和日志

### 服务切换流程

1. 完成代码与验证
2. 写入并启用 `systemd` 用户服务
3. 停掉当前 `screen`/旧进程
4. 用新的守护方式启动
5. 验证 `/status`、`/result`、日志、自恢复能力

### 约束

运行方式改变不影响接口，不改变输出。

## 验收标准

实施完成后，以下指标应成为验收依据：

### 推荐质量

- `rank1` 明显减少远高于市场却价值很差的候选
- `rank1` 更贴合真实路径价值
- `fUSD` 在默认场景中更稳定地主导高位推荐
- `fUST` 只在确有优势时上位

### 路径效果

- 新排序在真实执行链路仿真中的最终价值优于旧排序
- 落入 `FRR` / `rank6` 的比例更合理
- 长周期高收益机会命中率提升

### 流动性反应

- 最近 `2h/6h/12h` 的市场变化能更快体现在排序中
- 对 `fUST` 爆发与退潮的识别更敏捷

### 兼容性

- `result` 格式完全不变
- `rank6` 仍固定
- 外部机器人可直接无缝继续读取

### 服务治理

- API 改为后台守护运行
- 具备开机自启动和异常自动恢复能力

## 风险与控制

### 风险 1：路径模型过于复杂

控制：

- 先在现有模块内实现
- 先做最小必要路径仿真
- 不引入独立服务

### 风险 2：流动性快信号过度敏感

控制：

- 快信号主导，但保留慢信号校正
- 对 `fUST` 额外保守

### 风险 3：长周期偏置过强，重新推高假机会

控制：

- `120d >= 12%` 强优先仅在安全乘子不过度惩罚时生效
- 高溢价、低流动性、长期失败要能压回去

### 风险 4：内部验证逻辑改动后样本短期不稳定

控制：

- 保留旧统计口径可对比
- 新增评估脚本做旧新策略对照

## 实施原则

1. 先写计划，再实现
2. 尽量少增实体
3. 先对齐目标定义，再调权重
4. 先确保内部验证和真实路径一致，再谈模型更准
5. 先做服务守护方案，再切换线上运行方式

## 实施前结论

本次优化的关键不在于“继续微调静态利率模型”，而在于：

- 把推荐目标改成真实执行路径价值
- 把流动性信号改得更快
- 把 `fUSD` 优先与 `120d >= 12%` 强优先显式编码
- 把验证和训练标签校准到真实外部机器人行为

只要这些目标定义不改正，再继续小幅调整当前 `weighted_score`，效果仍会受限。
